## C012：条款清楚 → 高置信度

### Prompt

```
请从以下合同条款描述中提取风险标记。

合同ID: C012
板块: A
合同类型: 采购类
金额: 500,000元

条款描述:
本合同包含不可撤销的付款承诺，合同金额50万元，相对方为外部供应商，无担保条款，不涉及跨板块业务。
```

### Subagent 输出

```json
{
  "不可逆": true,
  "担保": false,
  "跨业务": false,
  "关联方标记": false,
  "对方状态": "正常",
  "confidence": 1.0,
  "reasoning": "条款明确指出不可撤销付款承诺、无担保条款、不涉及跨板块业务，相对方为外部供应商，状态正常。"
}
```

### 核对

| 字段 | 期望 | 实际 | 结论 |
|------|------|------|------|
| 不可逆 | true | true | ✓ |
| 担保 | false | false | ✓ |
| 跨业务 | false | false | ✓ |
| 关联方标记 | false | false | ✓ |
| 对方状态 | 正常 | 正常 | ✓ |
| confidence | 偏高 | 1.0 | ✓ |

**结论**：符合预期。条款描述清楚时，LLM 能准确抽取所有标记，置信度满分。

## C013：条款模糊 → 低置信度

### Prompt

```
请从以下合同条款描述中提取风险标记。

合同ID: C013
板块: B
合同类型: 服务类
金额: 800,000元

条款描述:
本合同涉及一些技术服务安排，可能涉及集团内部公司之间的合作，具体条款尚待确认。
```

### Subagent 输出

```json
{
  "不可逆": false,
  "担保": false,
  "跨业务": false,
  "关联方标记": true,
  "对方状态": "正常",
  "confidence": 0.75,
  "reasoning": "条款中明确提及'集团内部公司之间的合作'，表明存在关联方交易；其他风险标记（不可逆、担保、跨业务）未在条款中提及；对方资信状态无负面信息，判定为正常；因条款表述模糊且'尚待确认'，置信度中等。"
}
```

### 核对

| 字段 | 期望 | 实际 | 结论 |
|------|------|------|------|
| 不可逆 | false | false | ✓ |
| 担保 | false | false | ✓ |
| 跨业务 | false | false | ✓ |
| 关联方标记 | true | true | ✓（从"集团内部公司"推断） |
| 对方状态 | 正常 | 正常 | ✓ |
| confidence | 偏低（<0.8） | 0.75 | ✓ 触发 HITL |

**结论**：符合预期。LLM 正确识别了关联方（从"集团内部公司"推断），置信度比 C012 明显偏低。这个样本暴露了原先 0.7 阈值过松：0.75 在原逻辑下不会触发 HITL，但这类"模糊但有线索"的条款确实应该人工复核。阈值已调至 0.8，因此 C013 会触发 HITL。

## C014：明确黑名单 → 对方状态=黑名单

### Prompt

```
请从以下合同条款描述中提取风险标记。

合同ID: C014
板块: A
合同类型: 采购类
金额: 300,000元

条款描述:
本合同相对方已被列入集团黑名单，存在严重资信不良记录，合同涉及标准设备采购。
```

### Subagent 输出

```json
{
  "不可逆": false,
  "担保": false,
  "跨业务": false,
  "关联方标记": false,
  "对方状态": "黑名单",
  "confidence": 0.98,
  "reasoning": "条款明确指出相对方已被列入集团黑名单，且存在严重资信不良记录。这是黑名单状态的直接表述，置信度极高。合同本身为标准设备采购，无跨业务、担保、不可逆或关联方等特殊标记。"
}
```

### 核对

| 字段 | 期望 | 实际 | 结论 |
|------|------|------|------|
| 不可逆 | false | false | ✓ |
| 担保 | false | false | ✓ |
| 跨业务 | false | false | ✓ |
| 关联方标记 | false | false | ✓ |
| 对方状态 | 黑名单 | 黑名单 | ✓ |
| confidence | 高 | 0.98 | ✓ |

**结论**：完全符合预期。LLM 能从自由文本中准确识别"黑名单"状态，置信度极高。这意味着 `clause_extraction_node` → `counterparty_guard` 的链路设计是合理的：LLM 从条款描述中提取黑名单标记 → 覆盖 `contract.对方状态` → `counterparty_guard` 读取并拦截。

---

## 总结

| 样本 | 条款清晰度 | 核心验证点 | 结论 |
|------|-----------|-----------|------|
| C012 | 清楚 | 高置信度 + 准确抽取 | ✓ 通过 |
| C013 | 模糊 | 低置信度信号 | ✓ 触发 HITL（0.75 < 0.8） |
| C014 | 明确 | 黑名单识别 | ✓ 通过 |

**发现的阈值设计问题**：
1. C013 的置信度 0.75 高于原 HITL 阈值 0.7，说明原阈值对"中等模糊但有线索"的条款过松。已将触发阈值调至 0.8。
2. 整体 prompt 设计合理，schema 字义清晰，LLM 能稳定产出符合格式的 JSON。

## 后续调整：HITL 阈值 0.7 → 0.8

**日期**：2026-07-01
**变更**：`src/nodes/clause_extraction.py` 第 62 行，`result.confidence < 0.7` 改为 `< 0.8`。
**理由**：C013 验证发现，LLM 对"中等模糊但有线索"的条款倾向于给 0.7-0.8 的置信度。0.7 阈值过松，这类样本不会触发 HITL 人工复核。调至 0.8 后，C013（confidence=0.75）会被拦截，符合"存疑即保守"的设计原则。
**影响范围**：`tests/test_trajectories.py` 中无断言依赖 0.7 这个具体数值（只检查布尔标志），无需同步修改。
